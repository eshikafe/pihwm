1.1 Preparation

Unpack the software and create a separate directory in which to build it:

 
tar jxf pihwm-2012-09-25.tar.bz2
cd pihwm-2012-09-25
mkdir bd
cd bd

Alternatively, the latest version software may be cloned from git:

 
git clone git://github.com/embecosm/pihwm.git
cd pihwm
mkdir bd
cd bd
[ < ] [ > ]   [ << ] [ Up ] [ >> ]         [Top] [Contents] [Index] [ ? ]

1.2 Configuring the Build

Configure the software using the configure script in the main directory.

 
../configure

There are several options available, many of which are standard to GNU configure scripts. Use configure --help to see all the options. The most useful is --prefix to specify a directory for installation of the library.

1.3 Building and Installing

Build the tool with:

 
make all

If you have DejaGNU installed, you can verify the tool as follows (otherwise omit this step):

 
make check

Install the tool with:

 
make install

This will install the library in prefix/lib, the demo programs in prefix/bin, headers in prefix/include and the documentation in prefix/share, where prefix is the prefix specified when configuring, if any.

The documentation may be created and installed in alternative formats (PDF, Postscript, DVI, HTML) with for example:

 
make pdf
make install-pdf

2.1 Library Usage

The library may be used by adding -lpihwm when compiling on the Raspberry Pi. Programs should include pihwm.h and any other hardware specific headers from the include directory as required. Note that if a prefix has been specified when configuring, it may be necessary to specify the location of the library (using the -L flag) and the headers (using the -I flag).

For example, running on the Raspberry Pi:

 
gcc -I/opt/pihwm/include demo.c -L/opt/pihwm/lib -l pihwm -o demo

The general API is found in the pihwm.h header, which should be included before any other pihwm header.

Defined constant: INPUT
Defined constant: IN

These are both #defined to the constant string "in".

Defined constant: OUTPUT
Defined constant: OUT

These are both #defined to the constant string "out".

Defined constant: HIGH
Defined constant: ON

These are both #defined to the constant string "1".

Defined constant: LOW
Defined constant: OFF

These are both #defined to the constant string "0".

Macro: debug (...)

If the constant DEBUG is defined, then this maps directly to printf(...), otherwise it is empty.

Macro: delay (d)

This is defined as a synonymn for sleep(d).

2.2.2 GPIO API (pihwm_gpio.h)

The following functions are defined for driving the GPIO interface.

Function: int gpio_init (int pin, char *dir)

Initialize the GPIO interface for the pin numbered pin on the Raspberry Pi P1 header in the direction specified by dir. dir should be either "in" or "out", for which the defined constants INPUT, IN, OUTPUT or OUT may be used.

Returns 1 on success and -1 on failure.

Function: int gpio_release (int pin)

The inverse of gpio_init. Frees the pin numbered pin on the Raspberry Pi P1 header, so it can be used for other purposes.

Returns 1 on success and -1 on failure.

Function: int gpio_write (int pin, char *val)

Set the pin numbered pin on the Raspberry Pi P1 header to the value specified by val. dir should be either "0" or "1". The defined constants LOW or OFF may be used instead of 0 and HIGH or ON instead of 1/

Returns 1 on success and -1 on failure.

Function: int gpio_read (int pin)

Read the value on the pin numbered pin on the Raspberry Pi P1 header.

On success, returns 0 or 1 depending on whether the pin is low or high respectively. On failure returns -1.

Function: int gpio_set_int (int pin, void (*isr) (int), char *mode)

Set int as the interrupt service routine (ISR) for pin numbered pin on the Raspberry Pi P1 header. mode should be one of the strings "rising", "falling" or "both" to indicate which edge(s) the ISR is to be triggered on.

The function isr is called whenever the edge specified occurs, receiving as argument the number of the pin which triggered the interrupt.

Returns 1 on success. Never fails!

Function: int gpio_clear_int (int pin)

Clears any interrupt service routine (ISR) set on the pin numbered pin on the Raspberry Pi P1 header. The ISR will have been set originally by a call to gpio_set_int.

Returns 0 on success. Never fails!

In addition to the standard functions, a number of convenience functions are provided, which follow the Arduino style.

Function: int pinMode (int pin, char *val)

This is a synonym for gpio_init/

Function: int digitalWrite (int pin, char *val)

This is a near synonym for gpio_write, except that "HIGH" is mapped to "1 and "LOW" is mapped to "0 before passing val.

Function: int digitalRead (int pin)

This is a synonym for gpio_read.

Function: int attachInterrupt (int pin, void (*isr) (int), char *mode)

This is a synonym for gpio_set_interrupt.

Function: int detachInterrupt (int pin)

This is a synonym for gpio_clear_int.

3.1 Setting up the Raspberry Pi

For all these examples we are going to be using the Raspberry Pi headless, that is without its own keyboard, mouse and monitor. We’ll connect to it over Ethenet using the secure shell (ssh) program from a terminal window on another computer.

All the examples will work just fine with the conventional setup with keyboard, mouse and monitor. However you’re quite likely to want to use these circuits to control things like model cars or small robots, and you really don’t want to have to carry around anything more than a battery.

3.1.1 Setting up the SD card

The Occidentalis image can be found at

http://learn.adafruit.com/adafruit-raspberry-pi-educational-linux-distro/occidentalis-v0-dot-2.

Full details of how to put this on the SD card (it is not just a matter of copying) for a range of operating systems (Windows, Linux, Mac) are given at

http://elinux.org/RPi_Easy_SD_Card_Setup.

These is the short instructions for Linux. Run them from a shell.

  1. Unzip the image
  2. Use df -h, then insert your SD card and rerun df -h. The new entries are the devices corresponding to your SD card. You’ll probably see something like ‘/dev/mmcblk0p1’ or ‘/dev/sdd1’. The ‘p1’ or ‘1’ are the partition number on the device, but we want to write the whole device, so we’ll need to refer to it as ‘/dev/mmcblk0’ or ‘/dev/sdd’
  3. We don’t want anything else trying to write the SD card, so we now unmount it (leave it inserted). Use
     
    umount /dev/mmcblk0

    or

     
    umount /dev/sdd1
  4. Now use dd to copy the image to the device. Use the unzipped filename of the distribution and the device name for the whole card, not just the partition. For example:
     
    dd bs=4M if=Occidentalis_v02.img of=/dev/mmcblk0

    Change the filename and device name as appropriate.

    Note. This can take a long time (5-15 mins). You can check on progress by doing to another shell and running:

     
    sudo pkill -USR1 -n -x dd
  5. The image probably won’t fill the entire SD card, particularly if it’s large card. To fix this, use gparted. Run this and select the SD device (for example /dev/mmcblk0 or /dev/sdd). You’ll see two partitions, one with a fat16 file system and one with an ext4 file system. Select the ext4 partition and use Partition>Resize/Move to expand it to the maximum available size. Then use Edit>Apply Operations to change the size.

You now have an Occidentalis Raspberry Pi image using all the space available on your SD card.

3.1.2 Setting up the Network

The Raspberry Pi Occidentalis SD card is set up by default to accept SSH. The Ethernet interface will use DHCP to request an appropriate IP address. You can then connect to that address. For example if the IP address allocated was 192.168.0.77, you could use:

 
ssh pi@192.168.0.77

However for this to work, you need to know which DHCP address has been allocated to your Raspberry Pi card. If you are just using your home router, you can easily look to see what the value is. But in a more secure network environment this may not be possible.

Any router can be set up, so that a particular Raspberry Pi (identified by its MAC Address) will always be allocated the same IP address. In a school environment it might be appropriate to set this up for a classroom set of boards, labelling each board with the IP address.

In a standalone environment, it is good to be able to work without a full routed network. This is particularly useful when using a laptop with a standalone environment. To make this change, two modifications are needed:

  1. On the Raspberry Pi edit the ‘/etc/networks/interfaces’ file. Comment out the lines referencing DHCP and setting up the WLAN (use the # character at the start of each line).

    Then add the following lines.

     
    iface eth0 inet static
address 192.168.2.2
network 255.255.255.0
gateway 192.168.2.1

    This sets the Raspberry Pi up to have an IP address of 192.168.2.2 and connect to a host machine with IP address 192.168.2.1 You can use other numbers, but it it good practice to start with 192.168, and the address and gateway must only differ in the last number.

  2. Then on the host machine (e.g. a laptop), edit the network connections for the fixed network. Almost certainly this is set to automatic. Change to manual and specify the network address as 192.168.2.1. Don’t specify a gateway.

    You should now be able to connect the Raspberry Pi to the host machine by Ethernet and use ssh to log in

     
    ssh pi@192.168.2.2
